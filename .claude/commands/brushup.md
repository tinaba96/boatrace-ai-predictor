# Linear チケット ブラッシュアップ

引数: $ARGUMENTS (Linear チケットID。例: BOAT-123)

## 手順

### 1. チケット情報の取得
Linear MCP を使って、チケット `$ARGUMENTS` の詳細情報を取得してください：
- タイトル
- 説明
- 優先度
- ラベル
- 関連チケット

### 2. 並列調査（Agent Teams）
以下の2つのエージェントを**並列で**起動してください：

**エージェントA: コードベース調査**
- チケットの要件に関連する既存コードを探索
- 現状の実装がどうなっているか把握
- 技術的な制約や考慮事項を洗い出す

**エージェントB: 関連情報の収集**
- 関連チケットや親チケットの内容を確認
- プロジェクトの既存パターンやルールを確認
- 類似の過去実装があれば参考にする

### 3. チケット分析
調査結果をもとに、以下の観点でチケットを分析してください：

**要件の明確さ**
- 曖昧な表現や不足している情報はないか
- 「何を」「なぜ」「どのように」が明確か
- エッジケースは考慮されているか

**Acceptance Criteria（AC）の評価**
- ACが存在するか？なければ提案する
- 各ACは検証可能（テスト可能）か
- 漏れているACはないか（エラーケース、モバイル対応、パフォーマンス等）

**スコープの適切さ**
- チケットが大きすぎないか（分割すべきか）
- 依存関係は明確か
- 見積もりに影響する隠れた作業はないか

### 4. ブラッシュアップ提案
分析結果をもとに、以下を提案してください：

1. **改善後のタイトル**（必要な場合のみ）
2. **改善後の説明文** — 要件を明確に記述
3. **Acceptance Criteria** — 検証可能な形式で列挙
   ```
   - [ ] 〇〇した場合、△△が表示されること
   - [ ] 〇〇のエラー時、△△のメッセージが表示されること
   ```
4. **技術メモ** — 実装時に参考になる情報（変更対象ファイル、注意点等）
5. **見積もり感** — S/M/L（Small: 数時間、Medium: 半日〜1日、Large: 2日以上）

### 5. ユーザー確認
提案内容をユーザーに提示し、承認・修正のフィードバックを得てください。

### 6. Linear チケット更新
ユーザーの承認後、Linear MCP でチケットを更新してください：
- 説明文を改善後の内容に更新
- ACをチケットに記載

## 注意事項
- ユーザーの承認なしにチケットを更新しないこと
- 技術的な判断が必要な場合はユーザーに確認すること
- 既存の説明文を完全に上書きせず、元の意図を尊重すること
