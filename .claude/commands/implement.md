# Linear チケット自動実装

引数: $ARGUMENTS (Linear チケットID。例: BOAT-123)

## 手順

### 1. チケット情報の取得
Linear MCP を使って、チケット `$ARGUMENTS` の詳細情報を取得してください：
- タイトル
- 説明（実装要件）
- 優先度
- ラベル
- 関連チケット

### 2. ブランチ作成
チケットのタイトルから適切なブランチ名を生成し、`feature/` プレフィックスでブランチを作成してください。
- 例: `feature/BOAT-123-add-prediction-filter`

### 3. 並列調査（Agent Teams）
以下の2つのエージェントを**並列で**起動してください：

**エージェントA: コードベース調査**
- チケットの要件に関連する既存コードを探索
- 変更が必要なファイルを特定
- 既存のパターン・規約を確認

**エージェントB: 影響範囲分析**
- 変更による影響範囲を分析
- 依存関係を確認
- 既存テストの有無を確認

### 4. 実装計画（Plan）
調査結果をもとに、**実装前に計画を提示**してください：
- 変更対象ファイル一覧
- 各ファイルで行う変更の概要
- 実装の順序（依存関係を考慮）
- リスクや懸念点

**ユーザーの承認を得てから**次のステップに進んでください。
計画に問題がある場合は、フィードバックを反映して計画を修正してください。

### 5. 実装
承認された計画をもとに実装を行ってください：
- 既存のコーディング規約（`.claude/rules/code-style.md`）に従う
- モバイルファーストで実装
- デザイントークン（`src/styles/design-tokens.css`）を使用

### 6. 並列検証（Agent Teams）
実装完了後、以下の2つのエージェントを**並列で**起動してください：

**エージェントC: ビルド検証**
- `npm run build` でビルドエラーがないか確認

**エージェントD: コード品質チェック**
- 実装がチケットの要件を満たしているか確認
- セキュリティ上の問題がないか確認
- パフォーマンスの懸念がないか確認

### 7. コミット & Linear 更新
- 変更をコミット（コミットメッセージに `$ARGUMENTS` を含める）
- Linear MCP でチケットのステータスを「進行中」→「レビュー待ち」に更新
- チケットにコメントで実装内容のサマリーを追加

### 8. PR 作成
`/create-pr` コマンドの要領で PR を作成してください。
PR の説明に Linear チケットへのリンクを含めてください。

## 注意事項
- 実装が大きすぎる場合は、ユーザーに分割を提案してください
- 不明な要件がある場合は、実装前にユーザーに確認してください
- テストファイルがある場合は、テストも更新してください
