## 📋 プロジェクト概要

**BoatAI** はボートレース予想AI サービスです。選手の過去データ、モーター性能、当地実績、展示タイムなどを分析し、3つの予想モデル（スタンダード、本命狙い、穴狙い）で舟券戦略を提案します。

- **公式サイト**: https://www.boat-ai.jp/
- **リポジトリ**: https://github.com/rhapsody0919/boatrace-ai-predictor
- **デプロイ先**: Vercel (www.boat-ai.jp)
- **開発言語**: JavaScript/React 19, Node.js 22
- **データベース**: Supabase (PostgreSQL)

## 🏗️ アーキテクチャ

### システム構成

1. **フロントエンド**: React 19 + Vite 7 (PWA対応)
2. **API層**: Vercel Edge Functions (キャッシュ・RPC中継)
3. **データベース**: Supabase (PostgreSQL + RPC関数)
4. **バッチ処理**: GitHub Actions (5分毎〜定時スクレイピング)

### スクレイピング フェーズ別処理量

| フェーズ | 実行条件 | 対象レース | 実行頻度 | Egress |
|---------|---------|-----------|---------|--------|
| 1. レース情報 | 60分前ウィンドウ対象あり | 1～数件 | 5分毎 | 小 |
| 2. オッズ | 複数ウィンドウ対象あり | 1～20件 | 5分毎 | 中 |
| 3. 展示データ | 30/15/10分前ウィンドウ対象 | 1～10件 | 5分毎 | 中 |
| 4. 結果 | 発走後5分以上 | 1～数件 | 5分毎 | 小 |
| 5. 予測リフレッシュ | データ更新時のみ | 更新分のみ | 不定期 | 中 |
| 6. 予測買い目オッズ | 発走60分以内 | 1～20件 | 毎回 | 小 |

### スクレイピング最適化ポイント

✅ **実装済み最適化**:
- 対象ウィンドウ外なら即終了（ウィンドウなしで DB アクセスゼロ）
- データ更新時のみ予測リフレッシュ（全予測計算を回避）
- 5分毎 オーケストレーターで単一エントリポイント化

## 📊 データフロー図（テキスト形式）

### スコア計算フロー

```
racer_aggregated_stats（Supabase）
  ├─ 全国勝率、当地勝率
  ├─ モーター2連率
  ├─ ボート2連率
  └─ 級別ボーナス
       ↓
generate-predictions.js
  ├─ baseScore = 全国勝率×12 + 当地勝率×8 + 
  │              モーター2率×0.5 + ボート2率×0.3 + 
  │              級別ボーナス + 展示ボーナス
  ├─ 枠番補正（1号艇ボーナス +10）
  ├─ 会場別ルール補正（ruleMatchService評価）
  ├─ 展開予測（展示ST → 逃げ/差し/捲り確率）
  └─ 荒れ度スコア計算（0-100）
       ↓
predictions テーブル (Supabase)
  ├─ model_id: 'standard' / 'aggressive' / 'conservative'
  ├─ top_pick / top_2nd / top_3rd
  ├─ confidence スコア
  └─ predicted_at
```

### 的中判定フロー

```
predictions + race_results を照合
  ├─ 単勝的中: AI本命が1着
  ├─ 複勝的中: AI本命が2着以内
  ├─ 3連複的中: AIトップ3が1-2-3着を含む（順不同）
  └─ 3連単的中: AIトップ3が1-2-3着と順序も一致
       ↓
accuracy_summary テーブル（集計）
  ├─ hits / trials / accuracy（的中率）
  └─ payout / total_wagered / recovery_rate（回収率）
```

## ⚠️ 既知の課題

---

#### 2. スクレイピング失敗時のエラーハンドリング

**影響**: レース情報欠落 → ユーザーが当日レースを見られない

**課題**:
- HTML構造変更・ネットワーク障害で突然のスクレイピング失敗
- 本番環境での即時検知・復旧メカニズムが不十分
- エラー通知機能なし

**改善案**:
- [ ] Slack/メール通知機能
- [ ] リトライロジック（指数バックオフ）
- [ ] フォールバック（前日データ一時表示等）

---

### 🟠 中優先度

#### 4. 予測精度のばらつき

**影響**: モデル別・会場別に精度が大きく異なる

**現状**:
- スタンダードモデル: 複勝的中率50-55%
- 穴狙いモデル: 3連複的中率8-12%（ばらつき大）

**課題**:
- VENUE_VOLATILITY_WEIGHTSが固定値
- 月別・季節別の変動に対応していない
- ターン予測v3が展示ST依存のため、展示データ欠落時に精度低下

**改善案**:
- [ ] 動的パラメータ調整（月単位での自動retuning）
- [ ] 確率キャリブレーション分析
- [ ] 会場別・時間帯別の季節係数導入

---

#### 7. コード整理・リファクタリング

- `src/utils/turnPrediction.js`: ハードコード値が多い
