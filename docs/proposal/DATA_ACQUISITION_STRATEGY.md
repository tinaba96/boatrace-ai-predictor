# データ取得戦略 - サーバー負荷と利用価値のバランス

## 前提条件

### ボートレース公式サイトの特性
- 1日最大24会場×12レース = **288レース**
- レース間隔: 約30分
- 各レースページへのアクセスが個別に必要

### データ公開タイミング
| データ種別 | 公開タイミング | 更新頻度 |
|-----------|---------------|----------|
| 出走表（選手情報） | 前日〜当日朝 | 変更なし |
| 天候情報 | レース前に随時更新 | 数時間ごと |
| 展示タイム・ST | レース約30分前（展示航走後） | 1回のみ |
| オッズ | 発売開始〜締切まで変動 | 数分ごと |
| 結果 | レース終了後（約5分） | 1回のみ |

---

## 現状の取得フロー

```
[毎時0分] GitHub Actions
    │
    ├─ scrape-to-json.js (出走表・天候)
    │   └─ 約300リクエスト/回
    │
    ├─ generate-predictions.js (AI予想生成)
    │   └─ Supabase書き込みのみ
    │
    └─ scrape-results.js (結果取得)
        └─ 約300リクエスト/回（完了済みはスキップ）
```

**1日あたり推定リクエスト数**: 約600〜1,000リクエスト

## データ別の取得価値分析

### 1. 天候情報

| 取得タイミング | 利用価値 | サーバー負荷 | 実現可能性 |
|---------------|---------|-------------|-----------|
| 朝1回 | 中（ナイター未取得） | 低 | ✅ 現状 |
| レース直前 | 高 | 高（288回追加） | ⚠️ 要検討 |
| 結果取得時に併せて | 高（確定値） | 低（追加なし） | ✅ 推奨 |

**推奨**: 結果スクレイピング時に天候情報も取得し、`race_conditions`を更新
- 追加負荷: なし
- 取得できるデータ: レース時点の確定天候

### 2. 展示タイム・ST

| 取得タイミング | 利用価値 | サーバー負荷 | 実現可能性 |
|---------------|---------|-------------|-----------|
| レース30分前 | 最高（予想に活用可） | 非常に高 | ❌ 非推奨 |
| 15分間隔で取得 | 高（予想に活用） | 中 | ✅ **実装済み** |
| 結果取得時 | 中（過去分析用） | 低 | ✅ 推奨 |

**実装済み**: `scrape-exhibition.yml` で15分間隔（JST 9-17時）にbeforeinfoから取得。
- `exhibition_data` テーブルに保存、予測生成で展示ボーナス・ST優位性計算に活用
- 展示タイミング問題の詳細は [DATA_SCRAPING_GAPS.md](../issues/DATA_SCRAPING_GAPS.md)

### 3. オッズ

| 取得タイミング | 利用価値 | サーバー負荷 | 実現可能性 |
|---------------|---------|-------------|-----------|
| 締切直前 | 最高（期待値計算） | 非常に高 | ❌ 非推奨 |
| 結果取得時（確定オッズ） | 高（回収率計算） | 低 | ✅ 推奨 |
| 取得しない | - | - | 現状 |

**推奨**: 結果ページの確定オッズを取得
- 現状、配当金額は取得済み（payout_win等）
- 単勝オッズ（倍率）は未取得 → 期待値分析に有用

### 4. 決まり手・進入コース

|---------------|---------|-------------|-----------|
| 結果取得時 | 高（傾向分析） | なし（同一ページ） | ✅ **実装済み** |

**実装済み**: `scrape-results.js` で `winning_technique` と `course_1〜6` を取得・保存中。

### 5. レースグレード

| 取得タイミング | 利用価値 | サーバー負荷 | 実現可能性 |
|---------------|---------|-------------|-----------|

**実装済み**: `scrape-to-json.js` で取得 → `race_conditions.race_grade` に保存。

## リアルタイム取得が必要な場合の代替案

### 案1: 公式API（存在する場合）
- ボートレース公式がAPIを提供していれば、スクレイピングより効率的
- 現時点で公式APIの存在は未確認

### 案2: 外部データプロバイダー
- ボートレースデータを提供するサードパーティサービス
- 例: 競艇データベース系サービス
- コストがかかるが、サーバー負荷なし

### 案3: ユーザー起点の取得
- ユーザーがレースページを閲覧した際にデータを取得
- フロントエンドからのリクエストは公式サイトへの直接アクセス
- サーバー側でのスクレイピング不要

## 補足: ユーザーにとっての「利用価値の高い期間」

| ユースケース | 必要なデータ | 取得タイミング |
|-------------|-------------|---------------|
| レース前の予想閲覧 | 出走表、AI予想、天候 | 当日朝で十分 |
| 賭け直前の判断 | 展示タイム、ST、オッズ | レース30分前が理想 |
| 結果確認 | 着順、配当 | レース後 |
| 過去分析・モデル改善 | 全データ | レース後で十分 |

**現実的な判断**:
- 賭け直前のリアルタイムデータは**負荷が高すぎる**
- 過去分析用データは**結果取得時に併せて取得**が効率的
- ユーザーへの価値提供は**AI予想の精度向上**（過去データ分析による）が現実的
