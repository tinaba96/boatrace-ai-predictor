# AI対話型分析機能の検討

> **ステータス**: 検討中（方向性の議論完了、実装方針は未決定）

## 背景・目的

競合サイト「ボートレース日和（kyoteibiyori.com）」が提供する分析・統計機能（出目分析、場別統計、決まり手データ、アラート機能など）を、BoatAIサイト上で **AI対話形式** で提供する。

従来の「フィルター+テーブル」UIではなく、「xxxの会場でyyyの天候の時、どの選手が最も期待値が高い？」のように自然言語で質問し、データに基づいた回答を得られるチャットインターフェースを構築する。

### 競合分析資料

詳細は `docs/proposal/competitor-kyoteibiyori/` を参照（全15ファイル）。

## 競合機能の優先度

競合サイトの分析・統計機能を、BoatAIの蓄積データで実現可能かどうかを含めて評価。

### 高優先度

| # | 機能 | 概要 | BoatAIデータで実現可能か |
|---|------|------|------------------------|
| 1 | ガチガチレース検索 | 1号艇逃げ率+2号艇逃し率で堅いレースを抽出 | YES: race_entries + race_results |
| 2 | 穴レース検索 | 1号艇低逃げ率+外枠高攻撃率で荒れるレースを抽出 | YES: 同上 |
| 3 | 出目分析 | 会場×コース×決まり手で着順パターンを集計 | PARTIAL: winning_technique 36%のみ |
| 4 | 枠別決まり手データ | コース別の逃げ/差し/まくり率 | PARTIAL: 同上 |
| 5 | 場別統計 | 天候・グレード条件付きの会場統計 | PARTIAL: race_conditions 19%のみ |

### 中優先度

| # | 機能 | 概要 | BoatAIデータで実現可能か |
|---|------|------|------------------------|
| 6 | 勝率ランキング | 会場×コース×級別×期間の選手ランキング | YES: race_entries集計 |
| 7 | 番組解析 | レース番号別の平均配当・万舟率・イン勝率 | YES: race_results集計 |
| 8 | イン逃げ信頼度判定 | 1号艇逃げ率+2号艇逃し率の合算値 | YES: race_results集計 |
| 9 | トップページランキング | 固い場/荒れている場/イン逃/万舟率 | YES: 当日のrace_results集計 |

### 低優先度

| # | 機能 | 概要 | BoatAIデータで実現可能か |
|---|------|------|------------------------|
| 10 | 売上データ分析 | 場別の売上推移 | NO: データ未収集 |
| 11 | モーター貢献ポイント | モーター性能と選手実力の差分 | PARTIAL: motor_2rate はあるが貢献度計算は未実装 |
| 12 | 選手勝率推移 | 上昇/下降選手一覧 | PARTIAL: racer_id 32%のみ |

### データギャップ

分析機能の一部は、現在のスクレイピングのデータカバレッジに制約がある。

| テーブル/フィールド | カバレッジ | 影響する機能 |
|---|---|---|
| race_conditions（天候・風・波） | 19% | 場別統計の天候フィルター |
| winning_technique（決まり手） | 36% | 出目分析、決まり手データ |
| racer_id（選手ID） | 32% | 選手勝率推移 |

詳細: `docs/issues/DATA_SCRAPING_GAPS.md`

## アーキテクチャ選択肢

4つの設計案を比較検討した。

### 案A: Dify（既存コンポーネント活用）

```
ユーザー → DifyChat.jsx (iframe) → Dify Cloud → Knowledge Base / LLM
```

- `src/components/DifyChat.jsx` が既に存在（未接続）
- Knowledge Baseに分析JSONをアップロード

| 項目 | 評価 |
|------|------|
| 実装工数 | 1-2日 |
| 月額コスト | Dify Professional $59/月 + LLMコスト |
| 機能カバー | **~20%**（静的データのみ、動的集計クエリ不可） |
| モバイル | iframe のタッチ操作に問題あり |

**評価: 不採用**。動的なDB集計クエリに対応できず、競合機能の大半を実現不可能。

### 案B: LLM API + Vercel Edge Functions（推奨）

```
ユーザー → React Chat UI → /api/chat (Vercel Edge) → LLM (tool-calling) → Supabase
```

- 新規 `/api/chat.js` Edge Functionを作成
- LLMのtool-calling機能で意図を解析→事前定義されたクエリ関数を実行
- 既存の `api/races/today.js` と同じパターンで実装

| 項目 | 評価 |
|------|------|
| 実装工数 | 3-4週 |
| 月額コスト | LLM $2-100/月（利用量次第） |
| 機能カバー | **~85%**（データギャップ以外は全て対応可能） |
| モバイル | ネイティブReact UIで最適化可能 |

**評価: 最有力候補**。全ての競合機能に対応可能で、既存インフラと統一されたスタック。

### 案C: Supabase Edge Functions + LLM

案Bと同等の機能だが、Deno runtime上で動作。

| 項目 | 評価 |
|------|------|
| 実装工数 | 3-4週 + Deno学習コスト |
| レイテンシ | DB近接で50-100ms高速 |

**評価: 不採用**。Node.js → Denoのランタイム分裂、デプロイパイプライン分離のデメリットが上回る。

### 案D: RAG（Retrieval Augmented Generation）

```
ユーザー → Embedding → pgvector類似検索 → LLM
```

| 項目 | 評価 |
|------|------|
| 実装工数 | 4-6週 |
| 適合性 | **低**（構造化データの集計には不向き） |

**評価: 不採用（単体では）**。競合機能は構造化データへの集計クエリが中心であり、文書検索のRAGは適さない。定性的な質問（「なぜ平和島は荒れやすい？」）への補助としては将来的に検討可能。

### アーキテクチャ比較サマリー

| | Dify | LLM+Vercel | Supabase Edge | RAG |
|---|---|---|---|---|
| 機能カバー | 20% | **85%** | 85% | 30% |
| 実装工数 | 最小 | 中 | 中+ | 大 |
| 月額コスト | $59+ | **$2-100** | $2-100 | $2-100 |
| スタック統一 | × | **○** | × | △ |
| モバイルUX | × | **○** | ○ | ○ |

## LLMコスト分析

### Claude Code Max ($100/月) は使えない

Claude Code Maxは開発者向けCLIツールの定額プラン。サイト訪問者向けのAPI呼び出しには使用不可。サイトに組み込むにはAnthropic API（従量課金）の別契約が必要。

| LLM | 入力単価 | 出力単価 | 1クエリ目安 | 1,000クエリ/月 | 無料枠 |
|-----|---------|---------|-----------|-------------|--------|
| **Gemini 2.0 Flash** | 無料枠あり | 無料枠あり | ~$0.001 | **$0（無料枠内）** | **15 RPM, 100万トークン/日** |
| Claude Haiku 3.5 | $1/1Mトークン | $5/1Mトークン | ~$0.002 | $2/月 | なし |

## ChatGPTとの差別化

### 本質: 差別化は「AIの賢さ」ではなく「独自データへの接続」

| | ChatGPT | BoatAI Chat |
|---|---------|-------------|
| ボートレースの一般知識 | ○ | ○ |
| 12,700件以上の過去レースデータ集計 | × | **○** |
| 33,500件の予測と結果に基づく実績分析 | × | **○** |
| 24会場34ルールの独自戦略パフォーマンス | × | **○** |
| 今日のレースのリアルタイム予測 | × | **○** |
| 会場×天候×枠番のクロス集計 | × | **○** |

### 具体例

**質問:** 「平和島で3号艇のまくりが決まりやすい条件は？」

| ChatGPT | BoatAI Chat |
|---------|-------------|
| 「一般的に追い風でモーターが好調な時にまくりが決まりやすいです」 | 「平和島の直近90日: 3号艇まくり率14.2%。**風速3m以上の時は21.8%に上昇**（サンプル45件）。モーター2連率50%以上の時は回収率187%」 |
| 一般論 | **実データに基づく具体的な数値** |

ユーザーがデータを手動でChatGPTにコピペすれば同等の分析は可能だが、BoatAI Chatは **データが常に接続済みでゼロ操作** で結果を得られる。

## 2つの機能モード

AI対話機能には、性質の異なる2つの利用パターンがある。

### 分析モード（競合機能の模倣）

```
ユーザー: 「平和島の3号艇まくり率は？」
    ↓
LLM → 意図解析 → Supabase集計クエリ実行 → 結果を自然言語で返答
    ↓
「平和島の直近90日: 3号艇まくり率14.2%。
 風速3m以上では21.8%に上昇（サンプル45件）」
```

| 項目 | 詳細 |
|------|------|
| データソース | **Supabase（蓄積済みの過去データ）** |
| 自前DB | **必須**（歴史データの集計はリアルタイムスクレイピングでは不可能） |
| 実装方式 | LLM tool-calling → 事前定義クエリ関数 → Supabase |
| 差別化 | ChatGPTにはできない独自データ分析 |

### 予想モード（LLMにレースデータを渡して予想）

```
ユーザー: 「今日の平和島5Rどう思う？」
    ↓
boatrace.jp/Open APIから出走表を取得 → LLMに丸ごと渡す
    ↓
「3号艇の選手はモーター好調でまくり展開が狙えます。
 買い目: 3-1-4, 3-1-2, 3-4-1（3点）」
```

| 項目 | 詳細 |
|------|------|
| データソース | **boatrace.jp / Boatrace Open API（都度取得）** |
| 自前DB | **不要**（当日の出走表データのみ使用） |
| 実装方式 | 出走表JSON取得 → LLMに直接渡す → 予想生成 |
| 差別化 | 既存ルールベース予測との併用で独自性あり |
| 参考 | https://zenn.dev/mahiguch/articles/c41b200fe8938b |

### モード比較

| | 分析モード | 予想モード |
|---|---|---|
| 自前DB | 必須 | 不要 |
| 競合との差別化 | 高（独自データ） | 中（LLM予想は他サイトでも可能） |
| 実装難易度 | 高（tool-calling + クエリ設計） | 低（データ取得 + プロンプト設計） |
| 既存予測との関係 | 補完（分析根拠を提供） | 競合（第4モデルとして併存可能） |

## 自前データ（Supabase）の要否

### 結論: 分析機能をやるなら必須

| やりたいこと | 自前DB必要？ | 理由 |
|---|---|---|
| 今日のレースを予想して | 不要 | boatrace.jpを都度スクレイピングでOK |
| 平和島の過去90日のまくり率は？ | **必要** | 歴史データの集計はリアルタイムスクレイピング不可 |
| 回収率100%超のルールを教えて | **必要** | 予測結果と実結果の突合データが必要 |
| 今日の堅いレースはどこ？ | △ | 当日分は都度取得可能だが判定閾値は過去データから算出 |

競合サイトの分析機能（出目分析、勝率ランキング、場別統計etc.）は **全て過去データの集計** であり、都度スクレイピングでは物理的に実現不可能。boatrace.jpは過去データの一括取得APIを提供していない。

現在BoatAIのSupabaseには12,700件以上のレースデータが蓄積されており、これは分析モードの基盤として十分に活用可能。

## 案Bの実装設計（参考）

推奨案Bを採用した場合の技術設計。

### tool-calling設計（10関数）

| ツール名 | 対応する競合機能 | Supabaseクエリ |
|---|---|---|
| `search_tight_races` | ガチガチレース検索 | race_entries + race_results: 1号艇A1級 & 高勝率 |
| `search_upset_races` | 穴レース検索 | 同上: 1号艇低勝率 + 外枠高級選手 |
| `analyze_winning_patterns` | 出目分析, 決まり手データ | race_results: GROUP BY rank1, winning_technique |
| `get_venue_stats` | 場別統計 | races + race_conditions + race_results: JOIN集計 |
| `get_player_rankings` | 勝率ランキング | race_entries: 選手別集計 |
| `analyze_race_program` | 番組解析 | race_results: レース番号別集計 |
| `get_venue_rules` | 独自ルール参照 | data/analysis/venue-XX/rules.json 読込 |
| `get_in_escape_reliability` | イン逃げ信頼度 | race_results: 1号艇逃げ率集計 |
| `get_today_predictions` | 今日の予測 | 既存RPC `get_today_races` |
| `get_model_accuracy` | モデル実績 | accuracy_stats テーブル |

### フロントエンド

- `src/components/DifyChat.jsx` をネイティブReactチャットコンポーネント `AiChat.jsx` に置換
- 既存デザイントークン（`src/styles/design-tokens.css`）を使用
- フローティングボタン + チャットウィンドウのパターンはDifyChat.cssから踏襲
- モバイル対応: `onClick` のみ使用（CLAUDE.md制約）

### 段階的展開案

| フェーズ | 内容 | 期間目安 |
|---|---|---|
| Phase 1 | MVP: 3ツール（venue_rules, today_predictions, model_accuracy）+ 基本チャットUI | 1週間 |
| Phase 2 | 分析ツール追加（tight_races, upset_races, winning_patterns, venue_stats） | 1-2週間 |
| Phase 3 | 高度な機能（player_rankings, race_program, streaming, 会話履歴） | 1-2週間 |
| Phase 4 | 予想モード追加（出走表→LLM直接予想） | 1週間 |
