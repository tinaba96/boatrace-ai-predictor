# ルール運用ガイド

会場別ルールの追加・削除・監視に関する運用ルール。

> **関連ドキュメント:**
> - [system-specification.md](../reference/system-specification.md) - 技術仕様
> - [analysis-guide.md](../reference/analysis-guide.md) - 分析ガイド（ルール発掘手順）

## 概要

### 関連ページ

| ページ | URL | 用途 |
|--------|-----|------|
| 今日のおすすめ | `/picks` | ユーザー向け。ルール適用レースを表示 |
| 管理者ダッシュボード | `/admin/rules` | 管理者向け。ルール成績の監視・判断 |

### ルール定義ファイル

```
src/services/ruleMatchService.js
```

全ルールは `VENUE_RULES` オブジェクトにハードコードされている。

## ルールのライフサイクル

```
[発掘] → [コードに追加] → [検証期間] → [本格運用] → [成績悪化] → [除外]
```

### 1. 発掘

**分析スクリプトで回収率100%超えのパターンを発見**

```bash
node scripts/analysis/find-profitable-strategies.js
```

**採用基準:**

| 指標 | 基準 |
|------|------|
| 回収率 | 100%以上（必須） |
| サンプル数 | 150以上（推奨） |
| 期間 | 1ヶ月以上のデータ |

### 2. コードに追加

`src/services/ruleMatchService.js` の該当会場の配列にルールを追加。

```javascript
// 例: 徳山（18）に新ルール追加
const TOKUYAMA_RULES = [
  // 既存ルール
  {
    id: 'TY18-EX001',
    patternName: 'TOKUYAMA-EXACTA-R712',
    description: '3連単：後半レース(7-12R)',
    betType: 'exacta',
    stats: { samples: 233, hits: 22, recovery: 127 },
    reliability: 'high',
    check: (pred, raceNo, conf, predSorted, has1) =>
      raceNo >= 7 && raceNo <= 12
  },
  // 新ルール追加
  {
    id: 'TY18-W001',
    patternName: 'TOKUYAMA-WIN-NEW',
    description: '新しい条件',
    betType: 'win',
    stats: { samples: 150, hits: 80, recovery: 105 },  // 発掘時の統計
    reliability: 'medium',  // 検証期間中は medium
    check: (pred, raceNo, conf, predSorted, has1) =>
      /* 条件 */
  }
]
```

**ルールID命名規則:**

| 会場 | プレフィックス | 例 |
|------|---------------|-----|
| 江戸川(03) | E03 | E03-W001 |
| 平和島(04) | HW04 | HW04-EX001 |
| 浜名湖(06) | H06 | H06-W001 |
| 蒲郡(07) | G07 | G07-P001 |
| 津(09) | TS09 | TS09-EX001 |
| 三国(10) | M10 | M10-W001 |
| びわこ(11) | B11 | B11-W001 |
| 鳴門(14) | N14 | N14-W001 |
| 丸亀(15) | R15 | R15-P001 |
| 児島(16) | K16 | K16-W001 |
| 宮島(17) | MY17 | MY17-EX001 |
| 徳山(18) | TY18 | TY18-EX001 |
| 芦屋(21) | AS21 | AS21-EX001 |
| 福岡(22) | F22 | F22-W001 |
| 大村(24) | OM24 | OM24-EX001 |

**種別サフィックス:**

| 種別 | サフィックス |
|------|-------------|
| 単勝 (win) | W |
| 複勝 (place) | P |
| 3連複 (trio) | T |
| 3連単 (exacta) | EX |

### 3. 検証期間

**追加後30日間は検証期間**

- `/admin/rules` で週1回成績をチェック
- `reliability: 'medium'` のまま運用
- サンプル10件以上貯まるまで「注目レース」には表示されない

### 4. 本格運用

**検証期間後、以下を満たせば本格運用:**

| 指標 | 基準 |
|------|------|
| 運用サンプル数 | 10件以上 |
| 運用回収率 | 100%以上を維持 |

→ `reliability: 'high'` に昇格（任意）
→ `/picks` の「注目レース」に自動表示される

### 5. 成績悪化 → 除外

**除外検討の基準:**

| 状態 | 対応 |
|------|------|
| 回収率90-99% | 様子見（2週間） |
| 回収率80-89% | 除外検討 |
| 回収率80%未満 | 即時除外 |
| 2週間連続で100%未満 | 除外 |

**除外方法:**

`src/services/ruleMatchService.js` から該当ルールを削除またはコメントアウト。

```javascript
// 除外: 回収率悪化のため (2026-02-05)
// {
//   id: 'TY18-W001',
//   ...
// }
```

## 定期チェック運用

### 週次チェック（毎週月曜推奨）

1. `/admin/rules` にアクセス
2. 「概要」タブで全ルールの回収率を確認
3. 回収率100%未満のルールをリストアップ
4. 2週間連続で悪いルールは除外検討

### 月次分析（月初推奨）

1. 全会場の統計を収集
   ```bash
   node scripts/analysis/collect-venue-stats.js
   ```
2. 新しい高回収率パターンを探索
3. 有望なパターンがあればルール追加

## 表示内容の整理

### /picks ページ

| 要素 | 表示対象 | 説明 |
|------|---------|------|
| サマリー | 全アクティブルール | 運用開始日からの累積成績 |
| 注目レース | 回収率100%超え＆サンプル10以上 | 特に自信のあるレース |
| 各会場レースカード | 全アクティブルール | 全てのルールマッチ |

### /admin/rules ページ

| タブ | 表示内容 |
|-----|---------|
| 概要 | 全ルールの成績テーブル、週別推移グラフ |
| 会場別 | 会場ごとのルール成績 |
| 本日 | 今日適用されたルール一覧 |
| 履歴 | 過去の適用履歴（日付範囲フィルタ） |

## サマリー計算の仕様

### 計算対象

- `VENUE_RULES` に定義されている**アクティブなルール**のみ
- 運用開始日（2026-01-16）以降のデータ
- 結果が確定したレースのみ

### ルール削除時の挙動

**削除したルールの過去成績は計算から除外される**

例:
```
削除前: 34ルール、投資125,900円、回収142,520円、回収率113%
削除後: 33ルール、投資120,000円、回収140,000円、回収率117%
```

→ サマリーは「現在アクティブなルールで運用開始日から賭けていたら」という仮想成績を表示

## 注意事項

### 用語に関する注意

**「レース数」の実態はルール適用数**

画面上の「レース数: 1285」は、実際には「ルール適用数（結果あり）」を意味する。

```
例: 江戸川 7R に3つのルールがマッチした場合
- ユニークなレース数: 1
- ルール適用数: 3  ← これが「レース数」として表示される
```

詳細は [system-specification.md](../reference/system-specification.md) の「指標の計算仕様」を参照。

---

1. **ルール追加/削除後は必ずデプロイが必要**
   ```bash
   git add src/services/ruleMatchService.js
   git commit -m "feat: ルールXXを追加"
   git push origin master
   ```

2. **発掘時の統計（stats）は参考値**
   - 運用成績は `/admin/rules` で確認
   - 発掘時と運用時で成績が乖離することがある

3. **ルールIDは一意であること**
   - 同じIDが複数存在するとバグの原因になる
