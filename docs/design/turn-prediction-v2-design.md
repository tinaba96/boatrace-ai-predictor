# 展開予測システム v2 - 設計書

## v1 からの変更点

v1は「コース別の平均的な統計値で1パターンだけ予測し、スコアには反映しない」だった。
v2は「選手個人の攻守データを使い、マッチアップで3パターンを算出し、その結果をAIスコアに組み込む」。

## 提供する機能

### 1. 選手個人の実績に基づく展開予測

v1では「1コースの選手は全国平均で55%逃げる」というコース別の統計値だけで予測していた。v2では `racer_aggregated_stats` に蓄積された選手個人のデータを使う。

- **平均ST**: この選手のスタートはどれくらい速いか
- **攻撃実績**: この選手がNコースから勝った時の決まり手分布
- **被攻撃実績**: この選手がNコースにいた時、どの決まり手で負けたか

### 2. 攻守マッチアップによる確率算出

v1は攻撃側の実績だけで判断していた。v2では攻撃側と防御側の相性を見る。

例: 「3コースの選手がまくり得意（攻撃率45%）」+「1コースの選手がまくりに弱い（被攻撃率30%）」
-> 幾何平均 `sqrt(0.45 x 0.30) = 0.37` を基本確率として、ST差・モーター性能で補正する。

### 3. 上位3パターンの提示

v1は最も確率が高い展開1つだけを表示していた。v2では上位3つの展開パターンをタブUIで表示する。タブを切り替えると対応するアニメーションが再生される。ユーザーは複数の展開シナリオを想定した舟券戦略を立てられる。

### 4. 展開予測のAIスコアへの反映

v1では展開予測は「表示するだけ」で、3モデルのスコア計算には使われていなかった。v2では展開予測の結果がスコアに直接影響する。

- **本命狙い**: 逃げパターンで予測された1コースの選手にボーナス大（逃げ=堅い展開）
- **穴狙い**: まくり・差しパターンで予測された外枠の選手にボーナス大（波乱の主役）
- **スタンダード**: 均等に反映

### 5. 展示データのAIスコアへの反映

展示タイム・展示STもこれまでスコア計算に使われていなかった。v2ではモデルごとに重み付けしてスコアに加算する。

- **穴狙い**: 展示データ重視（機材のセッティングが重要）
- **本命狙い**: 展示データ控えめ（過去実績を信頼する）

## 技術設計

### データ基盤

#### DBカラム追加

`racer_aggregated_stats` に以下を追加:

- `defense_distribution` (JSONB): コース別被攻撃分布
- `course_race_counts` (JSONB): コース別出走数・勝数

マイグレーション: `docs/db-migration/add-defense-distribution.sql`

#### 集計関数（`scripts/analysis/aggregate-racer-stats.js`）

- `calculateDefenseDistribution()`: 選手がNコースにいた時に負けた決まり手の分布
- `calculateCourseRaceCounts()`: コース別の出走数と勝数

### 予測アルゴリズム (`predictFirstMarkV2`)

#### Step 1: ST予測
```
predictedST = exhibitionST * 0.55 + avgST * 0.35 + DEFAULT * 0.10
```
欠損時は利用可能なデータでフォールバック。

#### Step 2: ST優位性マトリクス
```
stAdvantage[i][j] = sigmoid((ST[j] - ST[i]) / 0.03)
```
0.5 = 互角、0.7 = i が 0.03s 速い（大きな優位）。

#### Step 3: 逃げ確率
- 個人の逃げ実績とデフォルト分布をベイズ縮小でブレンド（50走で個人データを完全信頼）
- ST差（2,3コースとの比較）、モーター性能で補正

#### Step 4: 攻撃パターン確率（2-6コース x 各決まり手）
- 攻撃側の実績 x 防御側（1コース）の被攻撃率 の幾何平均
- まくり系はST優位が必要、差しは控えめに反映
- モーターZスコアで補正
- コース出走数でのベイズ縮小

#### Step 5: 正規化 -> 上位3パターン選出
全パターン（逃げ含む）を正規化して合計1.0に。確率降順でソートし上位3つを返却。

### スコア統合

#### 展開予測ボーナス (`calculateTurnBonus`)

上位3パターンのうち、そのボートが勝者となるパターンの確率に応じてボーナス加算。

| モデル | 逃げパターン | その他パターン |
|--------|-------------|---------------|
| スタンダード | prob x 300 | prob x 300 |
| 本命狙い | prob x 500 | prob x 150 |
| 穴狙い | prob x 100 | prob x 400 |

#### 展示データボーナス (`calculateExhibitionBonus`)

| 要素 | スタンダード | 本命狙い | 穴狙い |
|------|-------------|---------|--------|
| 展示タイム差 | x50 | x30 | x80 |
| 展示ST | x40 | x25 | x60 |

### フロントエンド

#### 3パターンタブUI

- `patterns` 配列から最大3つのタブを生成
- タブ切り替えで `animKey` をインクリメント → アニメーション再生
- 旧フォーマット（`patterns` なし）も1パターンで正常表示（後方互換）

#### データ取得の互換性

`supabaseDataService.js` で新旧フォーマットを透過的にハンドル:
- 新: `patterns` 配列をそのまま使用
- 旧: `technique/winnerCourse/probability` から1要素の `patterns` 配列を生成

## 注意点

- **データ疎度**: 出走数が少ないコースはベイズ縮小でデフォルト分布にブレンド（50走で個人データを完全信頼）
- **スコアバランス**: 展開ボーナス（最大300-500）と展示ボーナス（最大50-80）は既存スコアレンジ（500-1500）に対して適度。チューニングが必要になる可能性あり
- **後方互換**: フロントエンドは旧フォーマット（patterns なし）も処理可能
- **パフォーマンス**: racer_statsの一括取得は1日あたり約900名。Supabase `.in()` の1000件制限を考慮し900件ずつチャンク処理
