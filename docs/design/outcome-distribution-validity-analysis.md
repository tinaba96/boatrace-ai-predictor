# 出目分布データの有効性検証 設計

## 概要

`outcome_distribution` テーブル（BOA-106 / migration 020）に格納された会場別の3連単出目分布が、実際の予想・投票判断における有用な指標として機能するかを統計的に検証する。

検証対象データ：

- `outcome_distribution`：会場別・1着〜3着の艇番組み合わせごとの過去90日出現回数・確率・平均配当
- `predictions`：モデル別の予測（top_pick / top_2nd / top_3rd, is_hit_*, payout_*）
- `race_results`：実績（rank1/2/3, payout_trifecta, course_1..6, winning_technique）
- `races` + `race_conditions`：レース属性（volatility, race_grade, 天候等）

検証の最終ゴールは「出目分布を使うことで予想精度・回収率が向上するか」を統計的有意性とともに判定することである。

---

## 検証の3階層

検証は粒度の異なる3つのレイヤで行う。

| レイヤ | 検証対象 | 主な問い |
|--------|----------|----------|
| L1：分布の安定性 | `outcome_distribution` 自体 | 過去90日の分布は将来も再現するか？ |
| L2：予測との整合性 | predictions ∩ outcome_distribution | AIの予測は実際に高頻度な出目に乗っているか？ |
| L3：投票戦略の有効性 | 分布ベースの買い目 vs 実績 | 分布を根拠に賭けると回収率が改善するか？ |

L1 → L2 → L3 の順で検証する。L1 が崩れていれば L2/L3 の議論は意味を持たない。

---

## L1：出目分布の安定性検証

### 目的

`outcome_distribution.probability` は過去90日のヒストリカルな出現率である。これが将来の出現率の推定量として機能するか（=分布が時間的に安定しているか）を検証する。

### 手法 1-A：分割サンプル一致度（Train/Test Split）

過去のレース結果を時系列で2分割する。

1. **学習期間**：T-180日 〜 T-90日（90日分）
2. **検証期間**：T-90日 〜 T 日（90日分）

学習期間で集計した分布を「予測分布」、検証期間の実出現率を「実分布」として比較する。

#### カイ二乗適合度検定

会場 v ごとに以下を計算する：

```
χ² = Σ_{i,j,k} (O_{ijk} - E_{ijk})² / E_{ijk}
```

- O_{ijk}：検証期間で出目 i-j-k が出現した回数
- E_{ijk}：検証期間の総レース数 × 学習期間の確率 p_{ijk}
- 自由度 df = (出現したユニーク出目数 - 1)

仮説：

- H0：学習期間と検証期間の分布は同一
- H1：分布は異なる

判定：

- p ≥ 0.05 → 分布は安定（出目分布は将来予測に使える）
- p < 0.05 → 分布は不安定（過去90日が将来90日を代表しない）

注意：E_{ijk} < 5 のセルは隣接セルにマージするか除外する（カイ二乗の前提条件）。

#### 補助指標：上位N出目の重複率

カイ二乗は全セルを見るが、実運用上は「上位パターンが当たるか」だけが重要なので、補助的に以下も計算する。

```
重複率_N = |Top_N(学習) ∩ Top_N(検証)| / N
```

N = 5, 10, 20 で算出。N=10 で 0.7 以上であれば実用上「分布は安定」と判断する。

### 手法 1-B：会場特性別の安定性

`venues.cluster`（in_strong / out_strong / balanced）別、`race_grade`（SG/G1/G2/G3/ippan）別、節日（series_day）別にカイ二乗を分割実行する。

期待される観察：

- `in_strong` 会場（徳山・住之江・大村）は 1-2-3 が支配的で安定が高い
- `balanced` 会場（戸田・江戸川）は分布が不安定で過去90日が将来を代表しない

### 手法 1-C：時系列ローリング検証

`last_updated` を 30日刻みでスライドさせて、過去6回分のスナップショットを取り、各時点での上位10出目が翌30日でどれだけ的中したかをプロットする。安定度の時間変化を見る。

## L2：予測と出目分布の整合性

### 目的

「AI が予測した3連単が、実際の出目分布のどこに位置しているか」を測定する。具体的には2つの問いがある。

- Q2-A：モデルの top_pick / top_2nd / top_3rd の組合せは、分布上位に集中するか分散するか？
- Q2-B：モデル別（standard / safeBet / upsetFocus）で分布カバー範囲はどう違うか？

### 手法 2-A：予測カバー率（Prediction Coverage Ratio, PCR）

各予測 p について、分布上での予測出目の順位（rank）と確率（probability）を引く。

```sql
SELECT p.model_id,
       p.race_id,
       od.probability AS pred_pattern_prob,
       RANK() OVER (PARTITION BY od.venue_code ORDER BY od.probability DESC)
         AS pred_pattern_rank
FROM predictions p
JOIN races r ON r.race_id = p.race_id
JOIN outcome_distribution od
  ON od.venue_code = r.venue_code
 AND od.first_boat  = p.top_pick
 AND od.second_boat = p.top_2nd
 AND od.third_boat  = p.top_3rd
WHERE p.is_shadow = FALSE;
```

集計：

- モデル別の平均 pred_pattern_rank
- 予測が分布 Top-10 に含まれた割合
- 予測が分布の母集団に1度も出現していない（rank 不明）割合

仮説：

- safeBet は平均 rank が低く（=分布上位寄り）かつ Top-10 ヒット率が高いはず
- upsetFocus は平均 rank が高く Top-10 ヒット率は低いはず
- 平均 rank が想定と乖離していれば、モデルの自己認識と実態がずれている

### 手法 2-B：予測確率と実出現率の相関

各レースで、モデルの予測（top_pick/2nd/3rd）に対応する出目分布の `probability` と、実際の的中 `is_hit_trio` の関係を見る。

ロジスティック回帰：

```
P(is_hit_trio = 1) = σ(β₀ + β₁ * pred_pattern_prob)
```

- β₁ > 0 かつ有意（p < 0.05）であれば、「出目分布で高確率な予測ほど実際に当たりやすい」が成立
- β₁ ≈ 0 なら、出目分布の確率は的中の予測力を持たない

`venue_code` をダミー変数として追加した拡張モデルも検証する。

### 手法 2-C：カイ二乗独立性検定（モデル × 分布位置）

クロス表を構築する。

|              | 分布Top-5 | Top-6〜20 | Top-20圏外 |
|--------------|----------|-----------|------------|
| standard     |   n11    |    n12    |    n13     |
| safeBet      |   n21    |    n22    |    n23     |
| upsetFocus   |   n31    |    n32    |    n33     |

χ² 独立性検定（df=4）で「モデルと分布位置は独立か」を検定する。p < 0.05 ならモデルごとに分布位置が有意に異なる（=モデルの個性が分布上に現れている）。

## L3：投票戦略としての有効性

最終的に「出目分布を使った戦略が回収率を改善するか」を測る。

### 手法 3-A：分布ベース買い目のバックテスト

戦略を定義：

- **戦略A**：会場ごとに分布 Top-N を均等買い（毎レース N点購入）
- **戦略B**：AI予測 ∩ 分布 Top-N の論理積（両方推す出目のみ買う）
- **戦略C**：AI予測 ∪ 分布 Top-N の論理和（カバレッジ重視）
- **ベースライン**：AI top_pick-top_2nd-top_3rd の1点買い

N = 3, 5, 10 でスイープする。

評価指標：

| 指標 | 計算式 |
|------|--------|
| 的中率 | hits / bets |
| 回収率 | Σ payout / Σ stake |
| 期待値 | (回収率 - 1) × stake |
| シャープレシオ風 | 平均利益 / 利益標準偏差 |

実装は `scripts/analysis/` に新規スクリプトとして配置（命名：`backtest-outcome-distribution-strategies.js`）。

### 手法 3-B：統計的有意性検定

戦略A〜C とベースラインの回収率差が有意かをブートストラップで検定する。

1. 検証期間のレース集合からランダムに同サイズでサンプリング（復元抽出、B=10000回）
2. 各サンプルで戦略Xとベースラインの回収率差を計算
3. 差の分布から95%信頼区間を構築
4. 信頼区間が0を含まない & 平均差 > 0 なら、戦略Xは有意にベースラインを上回る

回収率は重尾分布（高配当が稀に出る）なので、t検定よりブートストラップが適切。

### 手法 3-C：会場別・舟券種別の効果分解

戦略の効果が会場や舟券種でどう異なるかを見る。

- 会場別：24会場 × 3戦略のヒートマップ（回収率改善幅）
- 舟券種別：3連単 / 3連複 / 2連単 / 2連複 で同様に検証
  - 3連単：top_pick + top_2nd + top_3rd の順序固定
  - 3連複：3艇の組合せのみ（順序無視）
  - 2連単：top_pick + top_2nd のみ
  - 2連複：top_pick と top_2nd の組合せ

3連複・2連系の検証は、`outcome_distribution` から 3連単分布をマージダウンして算出する。

```sql
-- 3連複への変換例
SELECT venue_code,
       LEAST(first_boat, second_boat, third_boat)  AS b1,
       -- 中央値: 並び替えた中の2番目
       SUM(probability) AS prob
FROM outcome_distribution
GROUP BY venue_code, b1, b2, b3;
```

## 検証の優先順位とマイルストーン

| ステップ | 内容 | 工数目安 | 完了基準 |
|----------|------|---------|---------|
| Step 1 | L1-A カイ二乗で全24会場の分布安定性を検証 | 1日 | 全会場の p値 と安定/不安定分類が出る |
| Step 2 | L1-B/C 会場特性別・時系列の安定性を検証 | 1日 | クラスタ別・グレード別の安定度ヒートマップ |
| Step 3 | L2-A/B 予測と分布の整合性を測定 | 1日 | モデル別 PCR ・ロジスティック回帰係数 |
| Step 4 | L2-C モデル×分布位置のカイ二乗独立性 | 0.5日 | モデルが分布上で個性を出しているか判定 |
| Step 5 | L3-A バックテスト（戦略A〜C） | 2日 | 戦略×N×会場×舟券種の回収率テーブル |
| Step 6 | L3-B ブートストラップ有意性検定 | 0.5日 | 戦略ごとの95%信頼区間 |
| Step 7 | L3-C 会場別・舟券種別の効果分解 | 1日 | ヒートマップ + 推奨戦略の確定 |

## 統計的判定基準まとめ

| 検証項目 | 採用基準 |
|----------|---------|
| L1-A カイ二乗 | p ≥ 0.05（分布安定）または Top-10 重複率 ≥ 0.7 |
| L1-B 会場別 | 24会場のうち15会場以上で L1-A をパス |
| L2-A PCR | safeBet の Top-10 ヒット率 ≥ 60% |
| L2-B ロジスティック | β₁ > 0 かつ p < 0.05 |
| L2-C カイ二乗独立性 | p < 0.05（モデル差あり）|
| L3-A バックテスト | 戦略 X の回収率 > ベースライン回収率 |
| L3-B 有意性 | 95%信頼区間の下限 > 0 |
| L3-C 効果分解 | 24会場中 12会場以上で改善 |

すべての基準を満たせば「出目分布は予想精度向上に有用」と結論する。L1のみパスして L3 で改善が見られない場合、出目分布は「分析・可視化用」としては有用だが「投票戦略用」としては不十分と判定する。

## 想定される落とし穴と対策

### サンプルサイズの問題

- 1会場90日で約500-700レース。6艇の3連単は120通りあるため、希少出目はゼロ件のセルが多数発生する
- 対策：カイ二乗ではセル併合（E ≥ 5）。検証はTop-N（N=5,10,20）の重複に絞る

### 季節性・節リズム

- 節（series_day）別に分布が異なる可能性。最終日（is_final_day）は荒れやすい等
- 対策：L1-B で `race_grade` と `series_day` を交絡因子として層別検定

### モデル更新の影響

- predictions は過去のモデルバージョンの結果が混在する可能性
- 対策：`models.version` でフィルタするか、評価期間を固定して新旧モデルが混ざらないようにする

### 配当の重尾性

- 平均回収率は1件の高配当に引っ張られる
- 対策：L3-B はブートストラップで分布全体を評価。中央値回収率も併記

### バックテストの look-ahead bias

- 検証時点の `outcome_distribution.probability` は最新版（直近90日）であり、過去レース時点でのスナップショットではない
- 対策：バックテスト用には `race_date - 90日 〜 race_date` で都度集計し直す（`scripts/analysis/` 内で動的に計算）
